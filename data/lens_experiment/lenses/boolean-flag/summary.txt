A semaphore is a synchronization construct used in concurrent programming to manage access to resources and facilitate communication between threads or processes. It functions primarily as an integer-based counter that tracks available permits for a specific operation. At its core, the system serves two distinct purposes: limiting the number of simultaneous consumers for a shared resource, such as a database connection pool, and acting as a signaling mechanism to coordinate the execution order of different tasks.

The system operates through two atomic operations, commonly referred to as "wait" and "release." When a thread attempts to acquire a permit, it executes the wait operation; if the internal counter is greater than zero, the counter is decremented and the thread proceeds. However, if the counter is zero, the requesting thread is blocked and placed into a waiting queue until a permit becomes available. Conversely, the release operation increments the counter and signals the next waiting thread in the queue to resume execution. Relationships between threads can be producer-consumer in nature, where one task signals a semaphore to notify another task that a specific event has occurred or data is ready for processing.

Several rules and constraints govern the behavior of semaphores to ensure system stability. Unlike a mutex, which enforces strict ownership—meaning only the thread that locked the resource can unlock it—a semaphore is generally non-custodial. This allowing any thread to release a permit, which is essential for signaling but can lead to logical errors if used improperly for resource protection. A significant edge case involves priority inversion, particularly in real-time operating systems, where a high-priority task is delayed by a low-priority task because the system manager lacks visibility into semaphore ownership. Additionally, developers must account for the specific queuing implementation of the semaphore, as the order in which waiting threads are released may be first-in-first-out or entirely random depending on the underlying environment. Over-releasing a semaphore, where the number of release calls exceeds the number of wait calls, can also lead to unintended behavior by artificially inflating the available permit count.
